# /servitor — Чистка кодовой базы

Найди мусор, классифицируй по риску, убери в выбранном режиме.

## Режимы утилизации

| Режим | Действие |
|-------|----------|
| 🧹 **sweep** *(по умолчанию)* | Перемести в `.servitor-trash/` — откатываемо |
| 🔥 **burn** | Безвозвратное удаление — требует явного «да» от пользователя через AskUserQuestion |
| 📋 **scan** | Только отчёт, ничего не трогать |

## Классификация мусора

### 🟢 Безопасный (можно sweep автоматически)
- Пустые файлы (0 байт)
- `*.bak`, `*.orig`, `*.tmp`, `*.swp`
- `console.log` / `print()` оставленные после дебага
- Импорты которые не используются (статически доказуемо)
- Закомментированный код блоками >5 строк

### 🟡 Средний риск (sweep с записью причины)
- Файлы-сироты: никто не импортирует, не входит в entry points
- Экспорты которые никто не импортирует (мёртвые API)
- `TODO` старше 3 месяцев (по git blame)
- Тесты которые `.skip` дольше месяца

### 🔴 Высокий риск (только показать, не трогать)
- Файлы которые не менялись > 1 года, но не очевидно мусор
- Утилитарные скрипты в `scripts/` и `tools/` (могут использоваться через CI)
- Конфиги (`.env.example`, `tsconfig.json`, dotfiles)
- Файлы с подозрительными именами но непонятной функцией

## Workflow

1. Построй карту проекта: entry points (package.json `main`, скрипты, exports)
2. Запусти статический анализ — Grep на использование каждого модуля
3. Группируй найденное по риску
4. В режиме `sweep` создавай `.servitor-trash/<timestamp>/` сохраняя структуру путей
5. Выдай отчёт: сколько файлов в каждой категории, что перенесено

## Запреты

- Никогда не удалять без режима `burn` + явного подтверждения
- Не трогать `.git/`, `node_modules/`, `dist/`, `build/` — это не «мусор», это артефакты
- Не удалять файлы из `.gitignore` — они и так не в репо

# /fix — Исправление ошибок

Найди и устрани ошибки. Классифицируй по сложности и обрабатывай по правилам.

## Классификация

| Тип | Примеры | Действие |
|-----|---------|----------|
| **Простые** | форматирование, неиспользуемые imports, опечатки в именах, пропущенный `await` | Автоматически — фиксь и переходи к следующей |
| **Средние** | неверный тип, лишний/недостающий параметр, undefined references | Фиксь + кратко покажи что и почему |
| **Сложные** | логика, архитектура, race conditions, утечки | Не фиксь молча — объясни проблему, дай 2-3 варианта, спроси |

## Workflow

1. Запусти параллельно: `tsc --noEmit` (или эквивалент), линтер, тесты — собери все ошибки в один список
2. Сгруппируй по типу и файлу
3. Простые — батчем, средние — по одной с пояснением, сложные — через AskUserQuestion
4. После фиксов — **повторный прогон** проверок. Если новые ошибки появились → откат + анализ почему фикс сломал что-то ещё
5. Не маскировать ошибки через `// @ts-ignore`, `eslint-disable`, `try { ... } catch {}` без записи причины

## Запреты

- Не «исправлять» работающий код потому что «выглядит странно»
- Не менять API публичных функций ради тишины линтера
- Не подавлять ошибки которые сигнализируют о реальной проблеме

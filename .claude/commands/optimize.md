# /optimize — Оптимизация производительности

Выбери режим по подкоманде или из контекста. Не оптимизировать наугад — сначала измерить.

## Режимы

### `/optimize speed`
- Найди O(n²) и хуже в hot paths
- Замени линейный поиск на Map/Set где k уникален
- Убери лишние проходы по коллекции (один `.reduce` вместо `.filter().map()`)
- Кэшируй чистые функции с дорогим вычислением (memoize)
- Параллелизируй I/O где это безопасно (`Promise.all`)

### `/optimize memory`
- Утечки: незакрытые handles, держащиеся ссылки в closures, забытые listeners
- Генераторы / iterators вместо построения полных массивов
- Streaming вместо загрузки всего файла в память
- WeakMap/WeakSet для caches привязанных к объектам

### `/optimize render` (React / React Native)
- `memo()` на компонентах в больших списках
- `useMemo` / `useCallback` только когда есть профайлер-доказательство
- `FlatList` / `SectionList` вместо `.map()` в `ScrollView`
- Стабильные `key` (не индекс)
- `getItemLayout` для больших однородных списков

### `/optimize bundle`
- Tree-shaking: именованные импорты вместо `import * as`
- Динамические импорты для редко используемых экранов
- Удалить неиспользуемые зависимости (depcheck)
- Замена тяжёлых либ на лёгкие альтернативы

### `/optimize tokens` (LLM)
- Промпты: убрать повторы, перенести стабильные блоки в system (cacheable)
- Включить prompt caching (Anthropic SDK / OpenRouter / Bedrock)
- Батчинг вызовов: один запрос с 10 chunks вместо 10 запросов
- Сжать структурированный ответ: JSON keys покороче, без пустых полей
- `max_tokens` ровно столько, сколько нужно — не «на всякий случай»

## Workflow

1. **Измерь baseline.** Без цифр оптимизация — гадание.
2. **Найди bottleneck** — профайлер / `console.time` / Chrome DevTools / py-spy
3. **Один фикс — одна проверка.** Не делай 5 правок сразу: непонятно что помогло
4. **Зафиксируй цифры до/после** в commit message

## Запрет на спекулятивную оптимизацию

Не «давай добавим memo на всё». Если не доказано через профайлер — не трогай.
